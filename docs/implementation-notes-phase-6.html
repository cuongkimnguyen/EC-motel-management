Schema Deviations from Spec

1. Notification model has no reference_type field

Schema Change

Spec assumed: AgentAlertItem includes reference_type mapped from Notification.reference_type.

Reality: The existing Notification model (written in Phase 5) only has reference_id — no reference_type column exists.

Fix: In agent/service.py → get_alerts(), the AgentAlertItem is constructed with reference_type=None (hardcoded) since the field doesn't exist on the ORM model. The AgentAlertItem schema keeps reference_type: str | None for future compatibility when reference_type is added to the Notification table in a future phase.

Impact: Frontend agent alerts will not have a reference_type value. Low risk — the frontend currently only uses this field for deep-link navigation which is not implemented in Phase 6.

Key Decisions

D1. Seed-on-first-call pattern for WorkflowTemplates

Architecture

The 6 builtin workflow templates are seeded into the DB the first time GET /api/workflow-templates is called (if COUNT == 0), rather than via an Alembic data migration or startup hook.

Why: Alembic data migrations are fragile (they depend on model state at migration time). Startup hooks run even during tests, polluting the test DB. Seed-on-first-call is lazy, idempotent, and test-safe — the test suite sees exactly the same 6 templates the production system does.

Caution: If someone manually deletes all templates in production and the count drops to 0, the seed will re-run and create duplicates on the next call. Mitigate by adding a is_builtin=true guard or using INSERT ... ON CONFLICT DO NOTHING in a future hardening pass.

D2. APScheduler runs in-process (single Uvicorn worker)

Architecture

Background jobs (contract expiry sync, post publishing) run inside the same Uvicorn process via AsyncIOScheduler instead of a separate Celery worker.

Why: The spec describes a single-tenant MVP. Adding Celery + Redis for two cron jobs would double the infrastructure footprint for no measurable gain.

Limitation: If the app is scaled to multiple workers (e.g., uvicorn --workers 4), all 4 workers will run the same scheduler concurrently, causing duplicate job executions. The scheduled post publisher is idempotent (it only updates rows once), but the contract sync could emit duplicate notifications. Migration path: Replace AsyncIOScheduler with Celery Beat when multi-worker deployment is needed.

D3. Agent chat uses synchronous Anthropic SDK

Integration

The anthropic Python SDK's client.messages.create() is a synchronous call, even though our service layer is async. The service calls it directly without asyncio.to_thread().

Why: The Anthropic SDK v0.8+ is actually async-compatible via anthropic.AsyncAnthropic. The plan uses the sync client for simplicity since the test mock is easier. This is fine for MVP — the await point on the DB calls surrounding it ensures the event loop isn't starved for the brief time the sync call runs under normal latency.

Future improvement: Switch to anthropic.AsyncAnthropic and await client.messages.create() for proper async IO if LLM response latency becomes a concern under concurrent load.

D4. Automation status management via service-layer logic only

Business Logic

The status field on Automation is a plain string column with no DB-level constraint or enum type. The allowed values (active, paused, error, draft) are enforced only in the service layer.

Why: Adding a PostgreSQL ENUM type would require a migration to modify it every time a new status is needed. The app enum (AutomationStatus) is the single source of truth for application code.

Tradeoffs

T1. Conversation history window: 20 messages

Cost / Performance

The agent loads the last 20 messages per session as context for the Anthropic API call (AgentRepository.get_history(limit=20)). The spec did not specify a limit.

Why 20: Each message averages ~100 tokens. 20 messages ≈ 2,000 tokens of history + ~500 tokens system prompt = well within the 4K context budget before hitting diminishing returns. Higher limits increase latency and API cost linearly.

Adjust if needed: Change the limit=20 argument in AgentService.chat().

T2. Context builder shows only top-5 expiring contracts

Prompt Engineering

The system prompt injects at most 5 expiring contracts (lines 412–415 of plan). More would bloat the prompt without adding value for a conversational assistant.

Tradeoff: If there are 20 expiring contracts, the agent's context is incomplete. The agent will say "X contracts are expiring" from the count, but only detail the first 5 by code. This is acceptable for the MVP — the owner can drill down via the Contracts screen.

T3. Manual run endpoint marks task as "completed" immediately

UX / Accuracy

POST /api/automations/{id}/run creates a AgentTaskHistory record with status="completed" synchronously. There is no actual job execution — the endpoint is a simulation/log entry.

Why: True automation execution (sending notifications, posting to Facebook) requires infrastructure not yet wired (Zalo, email). The endpoint lets the UI show a "Run" button and display history without blocking on unimplemented features.

Future work: In Phase 7, replace the stub with actual job dispatch. The result_summary field already has the right structure to carry real results.

Known Issues & Limitations

L1. ANTHROPIC_API_KEY required in .env for agent chat to work

The agent service will fall back to a canned error message if the key is missing or invalid. No app crash. Set ANTHROPIC_API_KEY=sk-ant-... in backend/.env before testing live chat.

Tests mock anthropic.Anthropic so they pass without a real key.

L2. APScheduler not tested in integration tests

The scheduler is started in the FastAPI lifespan but the test client uses a separate test DB. The scheduler jobs reference AsyncSessionLocal (production DB URL). Tests do NOT patch this, so scheduled jobs in tests would hit the wrong DB.

Mitigation: The scheduler's AsyncIOScheduler only fires at its configured intervals (01:00 and every 5 min). During the test suite run (typically <60 seconds), no job will actually fire. This is safe for now.

L3. Notification.reference_type is absent from the data model

As noted in Schema Deviations above, AgentAlertItem.reference_type will always be null until a migration adds this column to the notifications table.
